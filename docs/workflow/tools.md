# `nekoai-tools` クレートのワークフロー

## 役割

`nekoai-tools` は、エージェントが利用するツール（Web 検索、コード実行、MCP 連携など）を登録・実行・権限制御するための拡張ポイントです。

現状コードベースでは `src/lib.rs` が空で、実装はこれから追加する段階です。ここでは **現状ワークフロー** と **導入時の推奨ワークフロー** を分けて定義します。

## 現状ワークフロー

1. `nekoai-agent` はツールレジストリを参照していない
2. LLM 推論はテキスト入出力のみで完結
3. `nekoai-tools` はビルド対象クレートとして存在するが、実行時の処理は持たない

## 導入時の推奨ワークフロー

### 1. 起動時初期化

1. `ToolRegistry` を生成
2. ビルトインツールを登録
3. 設定ファイルから許可/禁止ルールを読み込み
4. （必要に応じて）MCP サーバーへ接続しツール定義を同期

### 2. 推論中のツール呼び出し

1. モデルがツール呼び出し要求を返す
2. `PermissionGuard` が実行可否を判定
3. 許可されれば対象ツールを実行
4. 実行結果をモデルへ返して推論を継続
5. タイムアウトや中断要求があれば `AbortHandle` で停止

### 3. ログ/監査

1. ツール名・引数・実行時間・結果/エラーを記録
2. 失敗時は安全なエラーメッセージへ正規化
3. 機密情報をログに残さないようマスク

## 推奨モジュール分割

`architecture.md` に沿って、以下の構成を推奨します。

- `registry.rs`: ツール登録と列挙
- `permission.rs`: 実行可否判定
- `abort.rs`: 実行中ツールの中断管理
- `builtin/*`: 内蔵ツール実装
- `mcp/*`: MCP クライアント・トランスポート

## `agent` との連携ワークフロー（導入後）

1. `AgentRuntime` 初期化時に `ToolRegistry` を注入
2. セッションスコープに応じて有効ツールセットを取得
3. Rig エージェント生成時にツール一覧を渡す
4. ツール実行イベントをログまたは EventBus に発行

## 実装時の注意点

- デフォルトは deny 寄りにする（明示許可されたツールのみ有効）
- `code_exec` 系はサンドボックス前提
- 外部 API 連携ツールはレート制御と再試行方針を明確化
- ツール失敗で会話全体を落とさず、回復可能なエラーとして扱う

## 連携ポイント

- `nekoai-agent`: ツール呼び出し実行主体
- `nekoai-config`: ツール許可設定・MCP 接続情報
- `nekoai-infra`: 監査ログ/将来のイベント配信
